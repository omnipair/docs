Omnipair uses an **Impact-Aware Dynamic Collateral Factor** derived from the $xy=k$ invariant, designed to minimize manipulation and insolvency in [GAMM](/technical-breakdown/generalized-automated-market-maker) pools by dynamically adjusting borrowing limits based on loan size, liquidity, and market conditions.

## The Risks with Fixed Collateral Factors

**1. Slippage Bypass via Borrowing**

If AMM slippage exceeds the remaining margin $(1 - CF)$, users can borrow instead of swap to bypass the price curve:

- By borrowing until liquidation, users drain one side of the pool
- This breaks the AMM invariant and leads to one-sided liquidity exhaustion

**2. EMA Lag Exploitation**

If liquidation logic uses delayed EMA prices, users can:

- Borrow at inflated EMA valuations before the EMA adjusts to falling spot prices
- LPs incur losses while the protocol remains unaware of the solvency gap

**3. Split Position Attacks**

With fixed CF and per-position enforcement, users can:

- Split collateral across multiple positions to bypass pool-level borrowing limits
- Aggregate borrow amounts exceeding what the pool can safely support

## Dynamic Collateral Factor

These vulnerabilities are solved by computing collateral factors using **simulated liquidation with price impact**.

### Virtual Reserves at Pessimistic Price

The protocol constructs hypothetical reserves at the **pessimistic EMA price** (not spot price):

$$
P_{\text{pessimistic}} = \min(P_{\text{directional}}, P_{\text{symmetric}})
$$

Virtual reserves are reconstructed at this pessimistic price while preserving the AMM invariant $k = x \cdot y$. This prevents manipulation by:

1. Using the double-EMA system's pessimistic price instead of manipulable spot price
2. Ensuring collateral is always valued conservatively

### Price Impact Calculation

The CF calculation uses the constant-product swap formula to simulate the price impact of liquidating collateral:

$$
\Delta x = \frac{\Delta y \cdot x}{y - \Delta y}
$$

This computes how much collateral would be needed to cover debt via an AMM swap, accounting for slippage. The function `calculate_amount_in()` simulates:

1. **Utilized collateral**: Collateral needed to cover existing pool debt
2. **Maximum allowed debt**: Total debt the pool can support given remaining reserves
3. **User's borrow limit**: `max_allowed_debt - existing_pool_debt`

This ensures the protocol can always recover debt by selling collateral through the AMM.

## Two Collateral Factor Values

The system returns two distinct collateral factors:

| CF Type | Description | Usage |
|---------|-------------|-------|
| **Liquidation CF** (`liquidation_cf_bps`) | Threshold at which position becomes liquidatable | Liquidation checks |
| **Borrow CF** (`max_allowed_cf_bps`) | Maximum CF for new borrows | Borrow limit calculation |

### LTV Buffer

A **5% safety buffer** separates the borrow limit from the liquidation threshold:

$$
\text{Borrow CF} = \text{Liquidation CF} \times (1 - 0.05)
$$

This means users can borrow up to 95% of their liquidation threshold, providing a buffer before liquidation triggers.

**Example with 80% Liquidation CF:**
- Liquidation CF: 80%
- Borrow CF: 80% × 0.95 = 76%
- Buffer: 4% of collateral value

## Applied Min CF Protection

Each user position stores `applied_min_cf_bps` — the CF at the time of borrowing. This protects users from unfair liquidation:

**How it works:**
- When a user borrows, the current CF is recorded
- If pool conditions later cause CF to decrease, users keep their original CF
- Effective CF used: `max(current_cf, applied_min_cf)`

**Updated on:**
- New borrows
- Adding collateral
- After partial liquidation

This ensures users aren't liquidated due to pool conditions changing after they borrowed.

## Fixed vs Dynamic CF

Pools can be configured with either:

**Fixed CF:** A constant collateral factor (e.g., 85%)
- Simpler, predictable
- Still uses pessimistic pricing for divergence protection:

$$
\text{CF}_{\text{effective}} = \min\left(\text{CF}_{\text{fixed}}, \text{CF}_{\text{fixed}} \cdot \frac{P_{\text{directional}}}{P_{\text{symmetric}}}\right)
$$

**Dynamic CF:** Calculated from AMM reserves and price impact
- Adapts to pool liquidity depth
- Capped at `MAX_COLLATERAL_FACTOR_BPS` (85%)

## Borrow Limit Calculation

Each borrow request dynamically computes:

```rust
let (borrow_limit, borrow_cf_bps, liquidation_cf_bps) = 
    get_max_debt_and_cf_bps_for_collateral(&pair, &collateral_token, collateral_amount);

let remaining_limit = borrow_limit - user_current_debt;
```

This ensures **safety across all pool conditions**, recalculating in real-time on every:

- `borrow()`
- `repay()`
- `swap()`
- `liquidation check`

## Summary

Omnipair's **Impact-Aware Dynamic Collateral Factor** replaces static risk assumptions with real-time, on-chain math that:

- **Simulates liquidation impact** using AMM swap math
- **Uses pessimistic EMA pricing** to prevent manipulation
- **Maintains a 5% LTV buffer** between borrowing and liquidation
- **Protects user positions** with `applied_min_cf_bps` tracking
- **Adapts to liquidity depth** and market conditions

This protects both borrowers and liquidity providers from insolvency risks, price manipulation attacks, and one-sided liquidity drain.
